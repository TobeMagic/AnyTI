# BTI Matrix Platform

## What This Is

一个以 BTI 风格人格测试为核心的矩阵化静态前端平台，先从 `WBTI` 职场行为类型测试切入，逐步扩展到恋爱、消费、学习、健身、旅游等多个测试方向。产品的核心不是单个测试爆发，而是通过共用引擎、独立数据、站内互荐与主站聚合，沉淀一个可持续扩张的低成本测试矩阵。

## Core Value

用一套可复用的静态测试引擎，持续低成本孵化多个高传播测试，并把流量沉淀回自有矩阵资产。

## Context

项目定位不是“做一个测试”，而是构建一个 BTI 测试矩阵平台。每个测试面向不同生活场景与传播节点：职场、恋爱、消费、学习、健身、旅游；单个测试可借势节日或季节节点获取脉冲流量，矩阵结构则负责承接与循环流量。

技术路线强调零成本、零运维、零复杂度：纯静态前端部署到 GitHub Pages；代码一套，数据分离；目录层面由 `shared/` 放通用引擎，每个测试子目录仅维护 `questions.json` 与 `personalities.json` 两份数据。这样既利于快速复制，也利于后续 SEO 聚合。

科学性设计方向已经明确：每个测试应采用 4-5 个连续谱维度，不使用简单二选一打分；每道题可对多个维度赋权；结果匹配采用用户得分向量与人格向量之间的余弦相似度；每个测试再设计 1-2 个隐藏人格，通过极端值或特殊答题路径触发，以制造传播彩蛋与稀缺感。

产品传播机制也已确定为必须内置：结果页需要 Canvas 海报导出能力，海报上有人格名、核心描述、二维码，并适配朋友圈/小红书内容尺寸；结果页底部要推荐其他测试，把单次测试消费转成矩阵级浏览。

现阶段优先级清晰：先完成 `WBTI` 作为模板，再复用到 `LBTI`、`STBI`、`SBTI`、`FBTI`、`TBTI`。后续会再讨论前端风格、具体人格角色体系和每个测试的内容细节。

## Constraints

- **Tech stack**: 纯静态前端 + GitHub Pages — 用户明确要求零运维、零托管成本。
- **Architecture**: 引擎共用、数据分离 — 后续扩测试时必须保持边际成本接近零。
- **Scope**: 当前只定高层产品骨架 — 前端风格、角色命名、文案细节延后讨论。
- **Launch strategy**: `WBTI` 优先 — 必须先把单个模板测试全链路跑通。
- **Sharing**: 结果海报为必选项 — 传播能力不是附加项，而是产品内建能力。
- **SEO / IA**: 目录与信息架构需支持后续矩阵化聚合 — 不能只为单一测试临时拼装。
